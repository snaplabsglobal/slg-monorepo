# 💡 如何记录产品构思（Ideas）

> **快速答案：** 不用发全部聊天记录，只需要给我要点！

---

## 🎯 你的问题：

> "如果我突然想起的点子，和 GPT 讨论之后，想要整理一下...
> 应该把全部聊天记录发给你吗？按哪一类文档分类？"

---

## ✅ 标准答案

### 1️⃣ 不用发全部聊天记录！

只需要告诉 CDO：
- **背景：** 为什么想到这个点子？（1-2 句话）
- **核心想法：** 简单描述（3-5 句话）
- **讨论要点：** 和 GPT 聊出的关键结论（3-5 条）
- **下一步：** 打算怎么处理这个想法

**用时：** 2-5 分钟打字说明

---

### 2️⃣ 分类：放在 ideas/（产品构思）

```
docs/ideas/YYYY-MM-DD_构思_[你的想法名称].md
```

**为什么不放在其他地方？**

| 不放在 | 原因 |
|--------|------|
| decisions/ | 还没决定要不要做，太早了 |
| knowledge-base/ | 不是知识，是想法 |
| architecture/ | 不是技术方案，是产品构思 |

---

## 📝 两种记录方式

### 方式 A：极简版（推荐，2 分钟）

直接告诉 CDO：

```

GPT 建议：先做最简单的 MVP，不要一开始就做复杂的 3D

下一步：先记下来，等 Q2 再评估要不要做
```

CDO 会帮你整理成完整的构思文档！

---

### 方式 B：自己填模板（5-15 分钟）

使用 `快速构思记录模板.md`：

1. 复制模板
2. 填写核心部分（背景、想法、下一步）
3. 选填其他部分（应用场景、技术可行性）
4. 保存到 `ideas/YYYY-MM-DD_构思_XXX.md`

## 📂 ideas/ 文件夹的作用

### 💡 存放内容：
- 灵光一现的点子
- 早期产品构思
- 还没决定要不要做的想法
- 可能未来有用的创意

### 🚫 不存放：
- 已经决定要做的功能 → 放 `decisions/产品_`
- 已经在做的功能说明 → 放 `knowledge-base/知识库_产品_`
- 技术实现方案 → 放 `architecture/架构_`

### 🔄 想法的演化路径：

```
ideas/ (构思阶段)
    ↓ 
决定评估可行性
    ↓
decisions/ (决策阶段)
    ↓
决定要做了
    ↓
architecture/ (技术方案) + decisions/产品_ (产品方案)
    ↓
做完了
    ↓
knowledge-base/知识库_产品_ (产品说明书)
```

## 💡 实战案例对比

### ❌ 错误做法：

**情况：** 想到一个点子

**行动：**
1. 和 GPT 聊了 2 小时
2. 把 2 小时聊天记录全部复制给 CDO
3. 说"帮我整理一下"

**问题：**
- CDO 要花很多时间阅读
- 聊天记录有很多无关内容
- CDO 不知道你要重点记录什么

---

### ✅ 正确做法：

**情况：** 想到一个点子

**行动：**
1. 和 GPT 聊了 2 小时
2. 自己提炼出 3-5 个关键要点
3. 用 2 分钟告诉 CDO：背景、想法、要点、下一步
4. CDO 快速生成完整文档

**优势：**
- 节省双方时间
- 文档更聚焦
- 你自己也更清楚想法的核心

## 💬 常见问题

**Q: 如果想法很模糊，也要记录吗？**  
A: 要！越模糊越要记，不然过两天就忘了。记录时可以注明"想法不成熟"。

**Q: 如果讨论了 2 小时，真的不用发全部记录？**  
A: 真的不用。你自己提炼要点，效果更好。如果实在想留全记录，可以把链接/截图作为附件。

**Q: 什么时候把构思文档移到 decisions/？**  
A: 当你决定"要认真评估这个想法"时，就可以移了。

**Q: 构思文档会不会太多？**  
A: 不怕多！好想法多总比丢失好。每季度可以整理一次，把不要的归档。
