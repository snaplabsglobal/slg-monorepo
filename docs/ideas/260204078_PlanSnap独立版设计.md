## 📋 核心判断

**PlanSnap做好以后，完全可以、而且应该单独卖。**

房产经纪、DIY homeowner、只想申请一次permit的contractor——这个判断非常准。

---

# Part 1：为什么PlanSnap天生"适合单独卖"

## 满足独立售卖的4个硬条件

### ✅ 1️⃣ 单次高价值、非高频

- 画一次
- 出一套permit图
- 完成使命

👉 非常适合：
- 一次性用户
- 非订阅用户

---

### ✅ 2️⃣ 明确交付物

**PlanSnap的交付物不是"使用体验"，而是：**

- Permit-friendly PDF
- 政府敢收的图纸
- 可解释的尺寸

👉 这对非contractor用户非常重要。

---

### ✅ 3️⃣ 用户不想学复杂软件

- 不会用AutoCAD
- 不想学SketchUp
- 更不可能学Revit

👉 PlanSnap的"线 → 墙 → PDF"逻辑正好填补这个空白。

---

### ✅ 4️⃣ 不需要Job Photos / LedgerSnap

**房产经纪：**
- 只关心"能不能交permit / MLS"

**DIY homeowner：**
- 只想合法施工

**一次性contractor：**
- 不想换工作系统

👉 他们根本不想要一个平台。

---

# Part 2：怎么卖才不会伤到Jobsite Snap

## ❌ 错误做法（一定要避开）

- 把PlanSnap当成Jobsite Snap的"子功能"
- 强制用户先买Core才能用PlanSnap
- 用同一套定价逻辑卖给所有人

👉 这样会：
- 直接损失大量一次性用户
- 限制PlanSnap的增长上限

---

## ✅ 正确架构（非常重要）

### 产品结构要这样分：

```
Jobsite Snap（平台）
├─ Core（Photos）
├─ + PlanSnap（订阅 Add-on）
└─ + LedgerSnap（订阅 Add-on）

PlanSnap（独立产品）
└─ Standalone PlanSnap
```

**PlanSnap是唯一一个：**
- 既是平台Add-on
- 又是独立产品

这是非常高级、也非常合理的产品结构。

# Part 10：共用内核的技术拆分

## 最终形态

```
                 ┌────────────────────────┐
                 │  Geometry Core Engine   │
                 │  (Line → Constraint →   │
                 │   Wall → PDF)           │
                 └───────────┬────────────┘
                             │
        ┌────────────────────┴────────────────────┐
        │                                         │
┌──────────────────────┐               ┌──────────────────────┐
│   PlanSnap (Standalone)│               │  Jobsite Snap +     │
│   One-off / Project    │               │  PlanSnap Add-on    │
└──────────────────────┘               └──────────────────────┘
```

👉 所有几何逻辑只存在一份  
👉 没有"平台版PlanSnap"和"独立版PlanSnap"的算法差异

---

## 必须"绝对共用"的内核

### 🧱 1️⃣ Geometry Core（绝对单一）

**只存在一个版本的：**

- Line-first drawing
- Snap / inference priority
- Dimension = constraint
- Insert Partition Wall（center split）
- 净尺寸补偿逻辑
- Wall join / corner fill
- PDF生成逻辑

**📌 这些是PlanSnap的灵魂**  
**📌 任何"为了平台/独立版特殊处理"的冲动，都要压住**

---

### 🧮 2️⃣ 数学与单位系统（绝不妥协）

- 内部统一mm
- UI支持imperial
- rounding / tolerance规则一致

👉 不允许出现：
> "为什么独立版画出来的尺寸跟平台版不一样？"

---

### 📄 3️⃣ PDF输出引擎（必须一致）

- Inspector不关心你是哪个产品
- 他只关心：尺寸、清晰度、注释

👉 **PDF是法律文件，不能分裂**

---

## 必须"拆开"的地方

### 🎭 1️⃣ 产品外壳（UI / Flow）

**PlanSnap（Standalone）：**
- 入口即画图
- 没有Job概念
- Flow是：Draw → Generate → Download → Done

**Jobsite Snap + PlanSnap：**
- 从Job进入
- 有历史
- 有上下文
- Flow是：Photos → Area → Plan → Revision

👉 **这是体验层，不是内核层**

---

### 🧩 2️⃣ 数据上下文（Context）

**几何对象相同，但"谁拥有它"不同**

**Standalone：**
```typescript
Plan {
  id
  geometry
  created_at
  expires_at? (optional)
}
```

**Platform：**
```typescript
Plan {
  id
  job_id
  geometry
  version
  created_at
}
```

**📌 geometry字段结构完全一样**  
**📌 context字段不同**

---

### 💳 3️⃣ 商业逻辑（计费 / 权限）

**这层必须彻底独立于几何内核**

**Standalone：**
- Project-based billing
- PDF导出 = 付费点

**Platform：**
- Subscription
- 权限由account / plan决定

👉 内核不关心你付没付钱  
👉 外壳决定能不能"点那个按钮"

---

## 工程上最容易翻车的3个点

### ❌ 坑1：复制一份内核代码"改一改"

- 短期快
- 长期灾难

**正确做法：**
- 一个package / module
- 两个consumer

---

### ❌ 坑2：Standalone做"简化版几何"

比如：
- 少一个snap
- 少一个约束

👉 **会直接毁掉你"政府敢收"的品牌**

---

### ❌ 坑3：为了平台强行污染内核

比如：
- 内核里出现job_id
- 内核逻辑依赖Photos

👉 **这是架构腐烂的开始**

---

## 推荐的代码层级结构（抽象版）

```
/packages
  /geometry-core
    /line
    /constraint
    /wall
    /pdf
    index.ts

/apps
  /plansnap-standalone
    ui/
    billing/
    plan-context.ts

  /jobsite-snap
    jobs/
    photos/
    plans/
    permissions/
```

**📌 geometry-core永远不import app级东西**  
**📌 app级只调用core**

---

## 产品层的"心理一致性"（CPO视角）

### 你一定要确保：

**用户用独立版觉得：**
> "这是专业工具，不是阉割版"

**平台用户觉得：**
> "这是同一个PlanSnap，只是多了上下文"

👉 **同一引擎 = 同一信任**
