# PlanSnap & IsoSnap 战略决策 - 执行摘要

> **Date:** 2026-02-02  
> **Decision:** 双产品线战略  
> **Status:** ✅ Approved

---

## 🎯 一句话战略

**从单一产品（ISO SNAP）升级到双产品线（PlanSnap + IsoSnap）**

```
共用技术栈 ≠ 共用产品心智模型
两款独立的"心智工具"，不是同一产品的两个模式
```

---

## 📊 产品对比表

| 维度       | PlanSnap              | IsoSnap        |
| -------- | --------------------- | -------------- |
| **目标用户** | 普通 contractor / owner | 管工 / 技工        |
| **视角**   | 2D 平面俯视               | 等轴 30°/45°     |
| **心智模型** | "快速表达 layout"         | "能看懂的施工草图"     |
| **核心对象** | 房间、墙体、门窗              | 管道、节点、fitting  |
| **交互方式** | 拖拽、拉伸、对齐              | 起点→方向→长度→终点    |
| **用途**   | 客户沟通、permit           | Inspector、工地沟通 |
| **市场**   | 广度（用户多）               | 深度（粘性高）        |
| **定价**   | $19/月（单品）             | $19/月（单品）      |
| **界面风格** | 像 Notability          | 像制图软件          |

**全家桶：** $49/月（两个产品）

---

## ⚠️ 三条底线（绝对不能违反）

### 1. CPO 的底线

```
如果一个功能让你犹豫"这是 PlanSnap 还是 IsoSnap 的功能"
→ 那这个功能现在不该存在
```

### 2. COO 的底线

```
创业早期最容易犯的错误：
"顺手加个功能" → 产品变成谁都用不明白的"怪胎"
```

### 3. 硬隔离原则

```
❌ 共享同一套 Tool Bar
❌ 共享同一套 Mouse Interaction
❌ 共享同一套快捷键逻辑
❌ 用 "mode / toggle" 切换 Plan / Iso
```

---

## ✅ 可以共用（刀柄）

```
✅ Canvas / WebGL 渲染引擎
✅ Snap / Grid / Node / Edge 基础模型
✅ IndexedDB / LocalStorage
✅ PWA 架构（离线可用）
✅ PDF / PNG 导出
✅ Auth / License / Payment
✅ Project / File 容器
```

---

## ❌ 必须分离（刀刃）

```
❌ Tool Bar
❌ Mouse Interaction
❌ 快捷键逻辑
❌ 产品特定功能

→ 两个产品 = 两套交互哲学
```

---

## 🚫 禁止清单（Phase 2/3，现在不做）

```
❌ Code compliance 检查
❌ 自动 vent / slope 判断
❌ 材料清单 / 价格计算
❌ 多人协作
❌ 实时云同步
❌ JSS 深度集成 UI
❌ 2D 自动转换 ISO（COO 特别警告）
```

## 💼 商业策略（COO）

### 双线练手

- **PlanSnap：** 练"广度"（用户量大，测试 Stripe + PWA）
- **IsoSnap：** 练"深度"（粘性高，测试付费意愿）

### 品牌独立

```
PlanSnap: plan.snaplabs.io
IsoSnap:  iso.snaplabs.io

→ 虽然共用账号，但推广完全独立
```

---

## ✅ 验收标准（COO）

### 1. 匿名试用无感切换

```
用户不登录画完 → 点保存 → 注册 → 图纸毫秒级同步
→ 不能丢数据
```

### 2. 支付网关极速响应

```
从点击"解锁导出" → Stripe 支付 → 下载 PDF
→ 整个过程 < 60 秒
```

### 3. PDF 导出工程级质量

```
PlanSnap 导出：像建筑师画的
IsoSnap 导出：像教科书上的
→ 这种"高级感"决定是否愿意付 $49
```
