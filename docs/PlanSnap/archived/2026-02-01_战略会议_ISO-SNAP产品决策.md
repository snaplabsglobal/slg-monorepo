## 📋 会议背景

### 当前情况
- **LedgerSnap (LS)** 开发中，但面临高复杂度和冷启动难题
- **JSS (Job Site System)** 规划中，定位为完整系统
- **团队压力：** 需要确定性和现金流
- **技术基础：** 已有 web/前端能力和工地场景理解

### 提出的新策略
CEO 提出：先开发 **ISO SNAP**（独立 isometric 绘图工具），作为前锋产品，然后引流到 JSS。

---

## ✅ CPO 的核心结论

### 战略评价
> **CPO 评分：CPO 级别的肯定**

**关键判断：**
```
这是典型的"前锋产品（Front-Door Product）"策略：
- 低摩擦进入
- 高专业密度  
- 为复杂系统（JSS）筛选高质量用户

本质上是"工具 → 系统"的阶梯式增长，而不是一口吃成 SaaS。
```

## 🎯 战略优势分析

### 1. ISO SNAP 的独特优势

**命中真实刚需：**
- ❌ 不是：ERP / 项目管理 / 会计系统
- ✅ 是：**"我现在就要画一张 Inspector / 客户看得懂的 isometric"**

**特点：**
- 工地
- 手机/iPad
- 即画即用
- "今天就能用"的工具

---

### 2. 绕开 LS 的冷启动难题

**LS 的难点：**
- 角色多（老板 / 会计 / 工人）
- 数据重（receipt / project / cost）
- 学习成本高

**ISO SNAP：**
- 单人
- 单任务
- 3 分钟上手

→ **完美的漏斗上游产品**

---

### 3. 专业身份入口

**不是：**
- ❌ 通用 drawing app（会被 GoodNotes 吊打）
- ❌ 通用 CAD（会被 AutoCAD 碾压）

**而是：**
- ✅ **"只为管工设计的 isometric + code-aware 标注"**
- 这是**身份产品（Plumber-first）**，不是工具产品

---

## ⚠️ 三个致命风险（CPO 警告）

### 风险 1：功能膨胀 ❌

**绝对不能加：**
- 多楼层
- 全套 plumbing design
- 复杂符号库

**后果：**
- 变成半吊子 CAD
- 丧失"快"的优势
- 和 JSS 职责重叠

**原则：**
> ISO SNAP 必须永远是：快、窄、狠

---

### 风险 2：用户认知偏差 ❌

**危险：**
- 用户只把 ISO SNAP 当画图工具
- 看不到 JSS 的价值
- 引流失败

**解决：**
> ISO SNAP 必须"天然不完整"，而不是完整解决方案

---

### 风险 3：团队注意力分裂 ❌

**危险：**
- ISO SNAP 有 roadmap
- JSS 也有 roadmap
- 陷入双产品拉扯

**原则：**
> ISO SNAP 不是"产品线"，而是 JSS 的"入口器官"

## 🏗️ 2–3 周 MVP 功能切片

### MVP 目标（一句话）

```
让一个普通管工，用手机 / iPad，
在 10 分钟内画出一张 Inspector 看得懂的 isometric 图，并能导出。
```

**不是：** 设计软件 / CAD / 教学工具

---

### MVP 功能边界（红线）

#### ❌ 明确不做

- ❌ 平面图（plan view）
- ❌ 多楼层
- ❌ 自动计算 code
- ❌ 项目管理
- ❌ 成本 / BOM
- ❌ 协作
- ❌ 云同步（先不做）

→ **只做"一次性输出一张图"**

---

### ✅ MVP 核心功能

#### Week 1: 核心画图能力
- ✅ Isometric 画布
  - 固定 isometric grid（不可旋转）
  - 三个方向：↑↓ vertical / ↘↙ drain
  - Snap to grid
- ✅ 三种管段工具
  - Drain（斜）
  - Vent（直）
  - Delete / Undo

#### Week 1–2: 极简 plumbing 语义
- ✅ 管径标注（必选）
  - 1-1/2" / 2" / 3" / 4"
  - 点一段 → 选尺寸 → 自动显示
- ✅ Fixture 快捷符号（只要 5 个）
  - WC / Lav / Sink / Shower / FD
  - **"挂名点"**，只起说明作用

#### Week 2: Inspector 友好导出
- ✅ 一键导出 PDF / Image
  - 黑白
  - 清晰线条
  - 带标题栏：
    - "Isometric Sketch – Not for Construction"
    - 日期
    - 用户名（可选）

#### Week 2–3: 极轻量加分项
- ✅ 快速标签（不是校验）
  - 添加文本：
    - "WC wet vented via lav"
    - "Emergency drain – no vent"
    - "Existing 4" SAN"

---

### 到 JSS 的"自然引流点"

**引流点 1：保存历史**
```
免费版：只能导出，不保存
弹一句："Save this sketch with photos & inspection notes in JSS"
```

**引流点 2：项目关联**
```
导出时提示："Attach this sketch to a project (JSS)"
```

→ **不是广告，是工作流不完整**

---

### 技术栈

- **前端：** Canvas / Konva / Fabric.js（非常够用）
- **移动端：** PWA or iPad Web first（不急原生）
- **存储：** Local only（MVP）

→ **不需要后端，也不需要登录**

---

### 时间线

| 阶段 | 时间 | 内容 |
|------|------|------|
| Week 1 | 1周 | Grid + 管段 + Undo/Clear |
| Week 2 | 1周 | 管径标注 + Fixture + 导出 |
| Week 3 | 1周 | 文本标签 + UX 打磨 + 真实测试 |

---

### 成功标准

**只看 3 件事：**
1. 一个管工能不能不问你，就画完
2. Inspector 看不看得懂
3. **有没有人问：能不能存下来？**

→ 只要第 3 条出现，JSS 引流就成立

## 🚨 关键边界问题：Code 检查 & 材料计算

### 为什么不做 Code 检查？

#### 根本原因：责任边界

**一旦做 Code 检查，暗示：**
```
"这个画法在 Code 上是 OK 的。"
```

**在 plumbing 世界意味着：**
- Inspector dispute
- Permit dispute
- 责任转移（哪怕法律上写了 disclaimer）

→ **从"画图工具" → "合规系统"的跃迁**

---

#### 3 个致命问题

**1. Code 不是规则集，是解释体系**
```
- wet vent 在不在上游？
- emergency drain 算不算 indirect？
- 这个 FD 要不要 vent？

→ Code 80% 是解释，不是 if-else
```

**2. Code 检查会吃掉 LS 定位**
```
如果 ISO SNAP 说"这个合规/不合规"
→ 用户会问："那我还要 LS 干嘛？"
```

**3. Code 检查 ≠ MVP**
```
必须回答：
- 哪个版本的 Code？
- 哪个 City？
- 哪个 Inspector？
- Existing / New？

→ 这是 6–12 个月的坑，不是 2–3 周
```

---

### 材料计算的风险

**看起来"安全"，但有隐性风险：**

**一旦给数量，用户就会"拿去用"：**
```
你算出：20 ft 3" PVC + 4 个 wye
→ 他下单
→ 他预制  
→ 他发现不对
→ 他不会怪自己画错
→ 他会怪："软件算错了"
```

**材料计算 = BOM：**
```
BOM 是什么？
- 报价
- 成本
- 项目管理
- 会计

→ 这是 LS 的核心腹地
```

---

### 可行折中方案

#### 🟡 方案 A：Code 提示（Hint），不是检查（Check）

**允许做：**
- UI 级提示（不判断对错）
- 语言永远是：
  - "Common practice: …"
  - "Typical residential setup: …"
  - "Consider venting…"

**不允许做：**
- ❌ ✔️ / ❌
- ❌ Pass / Fail
- ❌ "Compliant / Non-compliant"

→ **你是在"陪他想"，不是"替他判"**

---

#### 🟡 方案 B：材料是"长度感知"，不是"清单计算"

**可以做：**
```
显示：
- "Approx. pipe length: ~18 ft"
- "Estimated fittings: 3–5 (varies by layout)"
```

**不能做：**
- ❌ 精确数量
- ❌ 导出 BOM
- ❌ 用于报价

→ **这是"画图反馈"，不是"施工准备"**

---

#### 🟢 最聪明的做法（强烈建议）

**把这两个需求全部留给 LS：**

```
在 ISO SNAP 里显示：
"Advanced code checks & material takeoff available in JSS."
```

**效果：**
- ✅ ISO SNAP 不背锅
- ✅ LS 一出场就是"升级"
- ✅ 用户心理预期非常干净
