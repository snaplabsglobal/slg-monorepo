# 📋 文档命名自查清单

> **使用场景：** 每次创建新文档时快速检查  
> **检查时间：** 30 秒  
> **目的：** 确保文档符合 CDO 规范

---

## ✅ 5 步快速检查

### 1️⃣ 选对文件夹了吗？

```
问自己：这个文档主要是什么类型？

□ 架构设计、数据库 Schema、系统框图
   → 放 architecture/

□ 功能决策、规范制定、项目规划、重构总结
   → 放 decisions/

□ Bug 修复、故障排查、问题解决
   → 放 troubleshooting/

□ 操作手册、快速参考、设计规范（长期不变）
   → 放 knowledge-base/
```

**不确定？** → 问 CDO（Claude Web AI）

---

### 2️⃣ 日期前缀正确吗？

```
□ 架构 / 决策 / 问题类文档：必须有日期前缀
   格式：YYYY-MM-DD_

□ 知识库类文档：不需要日期前缀
   因为：长期参考，不按时间排序

示例：
✅ 2026-02-01_决策_AI协作规范.md
✅ 知识库_部署流程手册.md
❌ AI协作规范_2026-02-01.md（日期放错位置）
❌ 2026-2-1_决策_AI协作规范.md（月份少了0）
```

---

### 3️⃣ 类型标签清楚吗？

```
□ 使用了正确的类型标签：
   架构_ / 决策_ / 问题_ / 知识库_

□ 标签与文件夹匹配：
   architecture/ 里的文档用「架构_」
   decisions/ 里的文档用「决策_」
   troubleshooting/ 里的文档用「问题_」
   knowledge-base/ 里的文档用「知识库_」

示例：
✅ architecture/2026-02-01_架构_数据库Schema.md
❌ architecture/2026-02-01_决策_数据库Schema.md（标签错了）
```

---

### 4️⃣ 描述够简洁吗？

```
□ 主题描述在 10 字以内
□ 能看懂是关于什么的
□ 优先使用中文
□ 保留必要的技术术语（SSO, API, ML）

示例：
✅ 2026-02-01_架构_Supabase-Schema重载.md
✅ 2026-02-01_决策_ML训练指南.md
❌ 2026-02-01_决策_关于我们如何实施机器学习模型训练的详细指南和最佳实践.md（太长）
❌ 2026-02-01_问题_bug.md（太模糊）
```

---

### 5️⃣ 文件名避坑了吗？

```
避免这些常见错误：

❌ 全大写：DATABASE_SCHEMA.md
   ✅ 改为：2026-02-01_架构_数据库Schema.md

❌ 空格太多：JSS LedgerSnap AI 协作规范 v1.0.md
   ✅ 改为：2026-02-01_决策_AI协作规范_v1.0.md

❌ 特殊字符：JSS: "Home Hero" 设计.md
   ✅ 改为：知识库_HomeHero设计要求.md

❌ 中英混杂无规律：JSS_协作_Collaboration_规范.md
   ✅ 改为：2026-02-01_决策_JSS协作规范.md
```

## 🚨 红灯警告

如果出现以下情况，**立即停止并找 CDO**：

1. 文档同时符合多个分类（如既是架构又是决策）
2. 不确定日期该用什么（过去的讨论？）
3. 描述怎么都超过 10 字
4. 文档名包含特殊符号（/ \ : * ? " < > |）

## 💡 快速命名公式

### 架构类
```
architecture/YYYY-MM-DD_架构_[系统名]-[模块名].md
```

### 决策类
```
decisions/YYYY-MM-DD_决策_[功能名/规范名].md
```

### 问题类
```
troubleshooting/YYYY-MM-DD_问题_[问题简述].md
```

### 知识库类
```
knowledge-base/知识库_[主题名].md
```

## 💬 常见问题

**Q: 我写的是一个会议纪要，放哪里？**  
A: 如果会议做了决策 → `decisions/YYYY-MM-DD_决策_XXX会议纪要.md`

**Q: 重构代码的总结文档算架构还是决策？**  
A: 决策类（因为是"为什么重构"而不是"系统如何设计"）

**Q: 临时笔记需要遵守规范吗？**  
A: 临时笔记可以随意命名，但整理后要改成规范格式

**Q: 能不能用英文命名？**  
A: 可以，但优先中文（团队主要语言），技术术语保留英文
