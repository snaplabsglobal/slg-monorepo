# CDO（文档官）协作快速指南

> **适用对象：** 所有与 Claude Web AI（CDO）协作的团队成员  
> **目的：** 5 分钟学会如何高效使用 CDO

---

## 1️⃣ 什么时候找 CDO？

### ✅ 应该找 CDO 的场景

| 场景 | 示例问法 | 期待输出 |
|------|---------|---------|
| **技术讨论需要记录** | "帮我把刚才 CTO 的回复整理成文档" | Markdown 技术文档 |
| **需要查历史决策** | "上次我们为什么决定用 pending 状态？" | 决策理由 + 相关文档链接 |
| **需要跨角色翻译** | "把这个技术方案用产品语言解释给 CPO" | 非技术人员能懂的说明 |
| **需要总结会议** | "总结一下今天的讨论要点" | 会议纪要 + 待办事项 |
| **需要更新文档** | "把这个决定加到规范文档里" | 更新后的 v1.x 版本 |

### ❌ 不应该找 CDO 的场景

| 场景 | 应该找谁 | 原因 |
|------|---------|------|
| "这个功能要不要做？" | Gemini 3（COO） | CDO 不做决策 |
| "这个架构有没有风险？" | Claude Code（CTO） | CDO 不做技术判断 |
| "会计能接受这个流程吗？" | GPT（CPO） | CDO 不懂业务规则 |
| "帮我写一段代码" | Cursor（工程） | CDO 不写生产代码 |

## 2️⃣ 如何与 CDO 高效协作？

### 提问模板

```
【背景】我们在讨论 XXX 问题
【讨论内容】[粘贴聊天记录 / 命令行输出 / 截图]
【需求】请帮我整理成文档，重点是 YYY

示例：
【背景】我们在讨论 409 冲突避免机制
【讨论内容】[粘贴 CTO 的两张命令行截图]
【需求】请帮我整理成 Markdown 技术文档，重点是状态流转逻辑
```

### 期待的回复时间
- 简单记录：立即（5 分钟内）
- 复杂文档：24 小时内
- 版本更新：当天完成

## 3️⃣ CDO 的标准输出格式

### 决策记录文档
```markdown
# [主题] 决策记录

## 背景
为什么讨论这个问题？

## 参与角色
- COO：XXX
- CTO：XXX
- CPO：XXX

## 决策内容

## 理由
为什么这么决定（技术 / 产品 / 优先级）

## 影响范围
这个决定会影响哪些模块 / 功能

## 后续行动
- [ ] 待办事项 1
- [ ] 待办事项 2
```

### 技术文档
```markdown
# [功能 / 模块] 技术说明

## 概述
一句话说明这是什么

## 架构图 / 流程图
[必要时包含]

## 核心逻辑
详细说明如何工作

## 边界条件
什么情况下会出问题

## 已知问题
目前的限制 / 技术债

## 参考资料
相关文档链接
```

## 4️⃣ 实战案例

### 案例 1：记录技术讨论

**用户：** "CTO 刚才在命令行里解释了状态管理机制，但我看不懂，能帮我整理吗？"

**CDO：**
1. 读取命令行截图
2. 提取核心逻辑（状态流转规则、数据来源、边界条件）
3. 转化为结构化 Markdown
4. 补充表格、流程图
5. 存入知识库

**产出：** 《状态管理与 409 避免机制.md》

---

### 案例 2：更新规范文档

**用户：** "我们决定加入 CDO 这个角色，帮我更新一下规范文档到 v1.1"

**CDO：**
1. 阅读现有 v1.0 规范
2. 在角色表中加入 CDO
3. 更新决策边界
4. 补充"统一问法"
5. 添加版本变更日志

**产出：**
- 《AI 协作与决策规范 v1.1.md》
- 《决策流程图 v1.1.md》

---

### 案例 3：跨角色翻译

**CPO（GPT）：** "CTO 说的'幂等性问题'是什么意思？"

**CDO：**
> "幂等性问题"在这里指：同一个操作执行多次，结果应该一样。
> 
> 具体到你们的场景：同一张收据上传两次，不应该在数据库里创建两条记录，
> 而应该识别出"这是重复上传"，直接返回第一次的结果。
>
> 这就像你买咖啡刷了两次信用卡，银行应该只扣一次钱，而不是扣两次。

**价值：** CPO 理解了技术概念，可以继续和 CTO 讨论产品方案

## 6️⃣ 如何查找 CDO 产出的文档？

### 文档存放位置
```
/docs/
├── decisions/          # 决策日志
│   └── 2025-02-01_决策_状态管理机制.md
├── architecture/       # 技术架构
│   └── 2025-02-01_架构_409冲突避免.md
├── troubleshooting/    # 问题排查
│   └── 2025-02-01_问题_OCR识别失败.md
└── knowledge-base/     # 知识库
    └── FAQ.md
```

### 文档命名规则
- **日期前缀：** YYYY-MM-DD（方便按时间排序）
- **类型标签：** 决策 / 架构 / 问题 / 会议
- **简短描述：** 10 字以内说清主题

## 7️⃣ 反馈与改进

如果你觉得 CDO 的文档有问题，可以：

1. **格式问题：** 直接告诉 CDO 改（如"表格太长，换成列表"）
2. **内容错误：** 指出哪里理解错了，CDO 会修正
3. **缺少信息：** 告诉 CDO 还需要补充什么

**示例：**
- ❌ "这文档写得不行"（CDO 不知道怎么改）
- ✅ "A3 部分的状态判定逻辑，需要加上边界条件的具体数值"
